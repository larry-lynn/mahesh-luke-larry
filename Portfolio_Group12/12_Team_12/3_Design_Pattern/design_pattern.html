Design Pattern

Design Patterns in the Compilers Course

The components of the compiler—the scanner, parser, symbol table, and semantic analyzer—were each by way of a specific pattern that students were asked to follow. In some cases, students were allowed to deviate from the patterns. As a result, some students may have chosen a different formal design pattern for portions of the compiler. If so, they appear below.

Specific Design Patterns

  • Facade Pattern

    • The facade pattern can be described as an intermediary between two or more different types of classes/programs. An advantage to using this design pattern is a centralized location for accessing different methods and types.
    • In our compiler project, we used the facade pattern to simulate the uPascal language using the Java language.
    • Exact line numbers of where this pattern can be found does not apply directly. This is because the scanner, parser, and semantic analyzer were designed with this pattern in mind and part of the project specifications.

  • Decorator Pattern

    • The decorator pattern is a way to abstract out specific details of how a class performs an opertaion.
    • A common example of our a program would make use of this is by iterating through a list. Different classes may iterate through their list differently because of list's ADT, but by using the decorator pattern, the program only needs to know that each class has a method to iterate.
    • Advantages of using this pattern are reusable code and less coupling of modules. What is meant by this is new objects that will make use of the decorator pattern can be added to the code with little modification to exisiting code.
    • A noteable spot where this project uses this design pattern is with the Symbol class. When the program needs to figure out what types of symbols we have when iterating, it simply goes over a general list of Symbols and casts when needed. This allows for dynamiclly casting of the Symbols and dynamic calls to class specific methods.

  • Singleton Pattern

    • Code that impelments the Singleton pattern essentially restricts a class to one instantiation.
    • This pattern was not implicitlly enforced in our project, but came about with the parser, semantic analyzer, and scanner. Each class is only created once in the course of the program, and as such thought it was appropriate to mention it.
    • An advantage to using this pattern is only one instantiation of the class is needed, and everything is accessed through it. Another advantage is this helps with concurrency via only one class is operating on data at a time, preventing possible errors.
